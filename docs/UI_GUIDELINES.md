# UI ガイドライン（RetroLoop デザイン標準）

このドキュメントは RetroLoop の UI 標準を「現状の実装ベース」で書き起こしたもの。
新しい画面・コンポーネントを作るときは、まずここに定義済みのトークン／クラス／パターンを使う。
足りない場合はアドホックに書かず、このドキュメントに追記してから実装する。

参照実装：`src/app/globals.css`（トークン＋共通クラス）、`src/app/layout.tsx`（フォント）、
`src/components/`（共通コンポーネント）。

---

## 0. デザインの方向性

- コンセプト：**「閉じるループ」**。整理して終わりではなく、問い返して改善が根付くまで追う。
- トーン：クールな紙色（`--paper`）＋インク（`--ink`）＋自社色 **iris（菫 `#5B4BE0`）**。
- 定番配色（ありがちな青緑グラデ等）は避け、**機能色は意味を持つときだけ**使う。
- ディスプレイ書体は一点使い（見出しのみ）。本文は素直なサンセリフ。

---

## 1. デザイントークン（`globals.css` の `:root`）

色は**必ずトークン経由**で使う（生の hex をコンポーネントに書かない）。
Tailwind v4 の `@theme inline` で `--color-*` にマップ済みなので、
`bg-surface` `text-ink` `border-line` `text-iris` のように Tailwind ユーティリティで参照できる。

### 下地・面・文字
| トークン | 用途 |
|---|---|
| `--paper` | ページ最下地（body 背景） |
| `--surface` | カード・入力欄の面 |
| `--surface-2` | 一段沈めた面（ホバー地・コード地・AI吹き出し） |
| `--ink` | 標準テキスト |
| `--muted` | 補助テキスト・ラベル |
| `--line` | 罫線・カード枠（弱） |
| `--line-strong` | 入力枠・区切り（強） |

### ブランド・アクセント
| トークン | 用途 |
|---|---|
| `--iris` / `--iris-ink` / `--iris-soft` | ブランド色（一次アクション・フォーカス・ユーザー吹き出し）／その上の文字／淡い面 |
| `--ember` | ループの起点を表す暖色アクセント（限定使用） |

### 機能色（意味を持つ色）
| トークン | 意味 |
|---|---|
| `--keep` | KPT: Keep |
| `--problem` | KPT: Problem |
| `--try` | KPT: Try |
| `--warn` | 「確認中」状態 |

### 付箋パステル（`--note-*`）
`yellow / green / blue / pink / purple / gray`。彩度控えめで `--ink` が読める明るさに統一。
機能色とは別系統（親しみ重視の背景色）。定義は `src/lib/note-colors.ts` と対。

### 形状
- 角丸：カード＝`--radius`(14px)、ボタン/入力＝10px、小ボタン/コード＝8px、丸バッジ＝999px。

### ダークモード
`@media (prefers-color-scheme: dark)` で**全トークンを差し替え済み**。
新色を足すときは**ライト／ダーク両方**に定義する（片方だけは不可）。

## 2. タイポグラフィ（`layout.tsx`）

| 役割 | 変数 | 書体 | 使いどころ |
|---|---|---|---|
| ディスプレイ | `--font-display` | Bricolage Grotesque (500/600/700) | 見出し（`font-display`）。一点使い |
| 本文 | `--font-sans` | Geist | 既定の本文 |
| モノ | `--font-mono` | Geist Mono | `.eyebrow` / `.badge` / コード |

- 言語は `<html lang="ja">`、`antialiased`。
- 見出しは `font-display font-semibold`。サイズは Tailwind（`text-2xl` 等）で指定。

---

## 3. 共通コンポーネントクラス（`globals.css` の `@layer components`）

| クラス | 用途 | 補足 |
|---|---|---|
| `.eyebrow` | セクション見出しの上に置く英字ラベル | mono・大文字・字間広め・`--muted`。構造に意味を与える |
| `.card` | コンテナ | `--surface` 面＋`--line` 枠＋`--radius`。パディングは利用側（`p-5` 等） |
| `.btn` | ボタン基底 | inline-flex 中央揃え・角丸10px・押下で微沈み |
| `.btn-sm` | 小ボタン | `.btn` と併用 |
| `.btn-primary` | 一次アクション | iris 背景。1画面に基本1つ |
| `.btn-ghost` | 二次アクション | 透明＋枠線。取り消し・補助操作 |
| `.field` | テキスト入力 | 枠＋フォーカスリング（iris）。`w-full` |
| `.badge` | 状態の小バッジ | 先頭に色ドット（`currentColor`）。KPT/判定の表示に |

### ボタン運用ルール
- **一次アクションは `.btn-primary` を1画面に1つ**を原則（迷わせない）。
- 補助・キャンセルは `.btn-ghost`。
- ⚠️ 現状 **danger / secondary バリアントは未定義**。破壊的操作（削除等）の赤ボタンが要るなら
  `.btn-danger` を本ドキュメントに追加してから実装する（アドホック禁止）。

## 4. レイアウトの規約

- ページ幅（`mx-auto max-w-* px-6`）の目安：
  | 用途 | 最大幅 |
  |---|---|
  | 認証・招待など単機能ページ | `max-w-sm` |
  | 標準コンテンツ（チーム詳細・一覧・AI結果） | `max-w-3xl` |
  | ボード（KPT 3レーン） | `max-w-7xl` |
- 共通ヘッダーは `AppHeader`（ログイン後）。中身は `max-w-3xl`。
- セクションは基本 `.card` で囲み、上に `.eyebrow` ラベル。
- 縦の間隔は `space-y-*` / `gap-*`（マージン直書きより Flex/Grid の gap を優先）。

---

## 5. フォームとサーバアクション

- 入力は `.field`、送信は `.btn`。
- 送信ボタンは二重送信を防ぐため、`useFormStatus` の `pending` で無効化する
  （参照：`src/app/signin/submit-button.tsx`）。
- ログイン後の戻り先は `callbackUrl`（相対パスのみ）を `safeCallbackUrl()` で検証してから使う
  （参照：`src/lib/safe-callback.ts`）。外部URLへのオープンリダイレクトを防ぐ。

### ⚠️ 既知の弱点（要改善・TODO）
現状フォームは**失敗時に無言 `return` するものが多く、ユーザーにフィードバックが出ない**
（招待バリデーション失敗、`devSignIn` の不正メール等）。以下は**まだ標準が無い**：
- フォームのバリデーションエラー表示（インライン）
- トースト／通知
- 成功フィードバック
- ローディング／スケルトン
- 空状態（empty state）コンポーネント
- モーダル／確認ダイアログ

これらは順次このドキュメントに標準を定義してから共通化する。

---

## 6. アクセシビリティ

- 現状フォーカスリングは `.field` のみ定義。**ボタンの `:focus-visible` は未整備（要改善）**。
- アイコンのみのリンク/ボタンには `aria-label` を付ける（例：`AppHeader` のホームリンク）。
- 装飾要素は `aria-hidden`（例：一覧の「→」）。
- 目標：キーボード操作・フォーカス可視化・コントラストの標準を今後ここに明文化する。

---

## 7. テスト用フック（`data-*` 属性）

E2E（Playwright）で参照するため、操作対象には `data-*` 属性を付ける慣習。
表示文言に依存せず要素を掴めるようにする。例：
`data-ai-chat` / `data-chat-input` / `data-chat-send` /
`data-invite-email-input` / `data-invite-submit` / `data-invite-link` /
`data-invite-revoke` / `data-invite-accept` / `data-member-row` / `data-invitation-id`。

新しいインタラクティブ要素を足すときは、対応する `data-*` を付けて spec から掴めるようにする。

---

## 8. 運用ルール（このドキュメントの使い方）

1. 新規 UI はまず本ドキュメントのトークン／クラス／パターンで組む。
2. 足りないものはアドホックに書かず、**まず本ドキュメントに標準を追記**してから実装する
   （過去に存在しない `btn-secondary` を書いてしまった反省）。
3. 新色は**ライト／ダーク両方**のトークンを定義する。
4. 破壊的操作・エラー表示・通知など「⚠️ 未整備」項目は、標準化のタイミングでここを更新する。
